# UX テスト計画 — truck-reception-app

作成日: 2026-04-22
目的: **ユーザーの使い勝手**に絞って網羅的に検証し、改善ポイントをスクリーンショット付きで記録する。

---

## 1. 基本方針

- **実機的に動かす**: プレビュー/本番で実際にクリック・入力して挙動を確認
- **視覚で残す**: 各ステップのスクリーンショット、問題箇所はハイライト
- **ユーザー目線**: 「技術的に動く」ではなく「迷わず使える」かを評価
- **優先順位**: ドライバー > キオスク > 管理画面（利用頻度・影響範囲順）
- **根拠を持つ指摘**: "なんとなく使いにくい" ではなく、タップ数・表示位置・色・コントラストなど具体値で評価

---

## 2. ペルソナ & 優先度

| # | ペルソナ | 利用画面 | デバイス | 頻度 | 優先度 |
|---|---------|---------|---------|------|--------|
| P1 | ドライバー（新規） | `/driver` | スマホ | 初回のみ | **高** |
| P2 | ドライバー（リピート） | `/driver`, `/driver/reserve`, `/driver/dashboard` | スマホ | 毎日 | **最高** |
| P3 | 機材センター スタッフ（キオスク操作） | `/kiosk` | 端末（タッチ） | 毎回来場時 | **高** |

| P5 | センター長（設定変更） | `/admin/centers`, `/admin/settings` | PC | 週1 | 低 |

---

## 3. 評価軸（UX Heuristics）

1. **Discoverability**: 何をすればいいか 3 秒で分かるか
2. **Efficiency**: 最小タップ数で完了できるか
3. **Error Prevention**: 間違えそうな操作にガードがあるか
4. **Recovery**: 間違えたとき戻せるか
5. **Feedback**: 操作結果が即座に分かるか（ローディング表示・成功/失敗メッセージ）
6. **Consistency**: 画面間でボタン配置・配色・用語が揃っているか
7. **Accessibility**: 高齢者・眼鏡着用者・グローブ着用者でも使えるか
8. **Performance**: 初回表示・ページ遷移の体感速度

## 5. シナリオ（golden path + edge case）

### P1: ドライバー（新規登録）

| # | ステップ | 評価軸 | 期待挙動 |
|---|---------|-------|---------|
| 1.1 | `/driver` アクセス | Discoverability | 電話番号入力が最初に目に入る |
| 1.2 | `090` ショートカット → 番号入力 | Efficiency | テンキーが大きく誤タップしにくい |
| 1.3 | 10桁未満で「次へ」 | Error Prevention | ボタンが押せない状態で視覚的にわかる |
| 1.4 | 11桁完成 → 次へ | Feedback | ローディング表示が出る |
| 1.5 | 電話番号確認画面 | Error Prevention | 大きく番号が出て誤登録防止 |
| 1.6 | プライバシーポリシー | Discoverability | 全文読める高さ、スクロール後ボタン活性 |
| 1.7 | 認証方法選択 | Consistency | 生体認証（推奨）とPINが同じカード高さ |
| 1.8 | 生体認証登録（Face ID / 指紋） | Feedback | 状態文言が遷移する |
| 1.9 | 完了 → ダッシュボード | Feedback | 成功トーストまたは遷移アニメーション |

**edge**:
- `0` で始まらない番号を入力
- `00` 連打
- 10桁で送信（固定電話扱い）
- 途中でブラウザ戻る
- PIN 入力ミス 3回連続
- Passkey キャンセル

### P2: ドライバー（リピート：予約作成）

| # | ステップ | 評価軸 | 期待 |
|---|---------|-------|------|
| 2.1 | `/driver` → 電話入力 → 生体認証 | Efficiency | 3 タップ以内でダッシュボード |
| 2.2 | ダッシュボード | Discoverability | 「予約する」ボタンが目立つ |
| 2.3 | `/driver/reserve` センター選択 | Efficiency | 表示対象センターのみ（showInDriverApp）|
| 2.4 | 日付選択 | Error Prevention | 過去日・休業日がグレーアウト |
| 2.5 | 時間帯選択 | Feedback | 満車スロットは「満車」表示 |
| 2.6 | 車両情報入力 | Efficiency | プロフィールから既定値補完 |
| 2.7 | 確認画面 | Consistency | 確認項目の順番が入力順と同じ |
| 2.8 | 予約確定 | Feedback | 予約番号表示 + QRコード |

**edge**:
- 同時刻に同じドライバーが 2 回予約
- 満車スロット選択
- ネットワーク切断中に確定タップ
- 日付未選択で「次へ」

### P3: キオスク受付

| # | ステップ | 評価軸 | 期待 |
|---|---------|-------|------|
| 3.1 | `/kiosk` トップ | Discoverability | 「受付開始」が画面中央に大きく |
| 3.2 | 電話番号入力 | Efficiency | 20 秒以内で入力完了（グローブ想定） |
| 3.3 | ドライバー選択（同番で複数） | Discoverability | 候補がリスト表示される |
| 3.4 | 車両番号入力 | Error Prevention | ナンバー形式（3-4 ブロック）のガイド |
| 3.5 | ナンバープレート 4 ステップ | Efficiency | 各ステップで戻る可能 |
| 3.6 | 予約照合 | Feedback | 予約ありなら自動で該当を選択 |
| 3.7 | 最終確認 | Error Prevention | 「取消」「印刷」が物理的に離れている |
| 3.8 | 印刷 → 完了画面 | Feedback | 15秒カウントダウン |
| 3.9 | タッチで一時停止 | Discoverability | カウントダウン停止が明示される |

**edge**:
- 電話番号入力途中で放置 5 分 → 自動リセット?
- 同じ車両で 1 分以内に 2 回受付
- 受付完了後すぐ戻るボタン

### P4: 管理画面（受付一覧）

| # | ステップ | 評価軸 | 期待 |
|---|---------|-------|------|
| 4.1 | ログイン | Efficiency | 2 タップで完了 |
| 4.2 | ダッシュボード KPI | Discoverability | 本日の受付件数・予約数が即座 |
| 4.3 | 受付一覧 | Efficiency | 自センター絞り込みが初期値 |
| 4.4 | 検索（日付・電話） | Feedback | 検索結果件数が即時表示 |
| 4.5 | 受付詳細 → 削除 | Error Prevention | 削除前に確認ダイアログ |
| 4.6 | CSV エクスポート | Discoverability | 「エクスポート」ボタンがすぐ見える |

## 6. スクリーンショット取得ポリシー

各シナリオで:
- **A. 正常系の各ステップ**（golden path）: 1 枚ずつ
- **B. 問題発見時**: その画面 + 1 つ前の画面
- **C. デバイス別**: P2 のみモバイル 3 幅で同一画面を並べる
- **D. アニメーション**: GIF or 連続 3 枚

保存: `docs/ux-screenshots/<persona>/<scenario-id>-<description>.png`

---

## 7. スコアリング

各シナリオを以下で採点:

| 評価 | スコア | 定義 |
|------|-------|------|
| 🟢 合格 | 0 | 問題なし |
| 🟡 改善余地 | 1-3 | 使えるが摩擦あり |
| 🔴 要修正 | 4-7 | ユーザーが迷う/間違える |
| 💀 致命 | 8-10 | 機能が壊れている / 完了できない |

合計スコアで優先度づけ。

## 8. 実行計画

### Phase 1: 静的レビュー（30分）
- 全画面のコードをスキャンし、明らかな UX 問題を先に拾う
  - フォントサイズ・タップ領域・コントラスト
  - ローディング状態の有無
  - エラーメッセージ文言

### Phase 2: 自動ウォークスルー（ドライバーシナリオ・2時間）
- 本番 URL に対して Chrome 自動操作で P1 → P2 をトレース
- 各ステップでスクリーンショット
- ネットワークタイミング計測（初回 TTFB、画面遷移時間）

### Phase 3: キオスク端末シミュレーション（1時間）
- タブレット幅で `/kiosk/*` を一通り実行
- グローブ前提のタップ領域検証（最小 48x48 CSS px 基準）

### Phase 4: 管理画面レビュー（1時間）
- デスクトップ幅で P4 を実行
- 検索・フィルタ・CSV 出力の実用性確認

### Phase 5: 所見レポート作成（30分）
- 発見した問題を Severity × Effort でマッピング
- 即修正できるもの（文言・色・余白）は同セッションで対応

### Phase 6: 修正・再検証（時間制限なし）
- 優先度 🔴 以上の問題を修正 → 再度 Phase 2-4 を該当箇所のみ再走

## 9. 実行時の前提

- **テストアカウント**: 510101 / パスワード 5101（岸和田）を使用
- **テストドライバー**: 本番 DB に影響を残さないため、テスト電話番号（例: 09099999999）で登録・即削除
- **ネットワーク**: 実測用に Chrome DevTools "Fast 3G" でも 1 シナリオ計測
- **ブラウザ**: Chromium (Claude Preview) をメインに、Safari 固有問題は別途メモ
